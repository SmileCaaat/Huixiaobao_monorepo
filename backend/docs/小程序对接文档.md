# 消防维保任务小程序对接文档

## 版本更新记录

### v1.5 (2026-03-16)
**新增功能：故障报修流程升级**

1. **场景升级**
   - 小程序端支持用户主动上报故障
   - 报修人可以持续查看自己上报的记录
   - 管理员/项目负责人可以查看当前单位全部报修，并可派发给指定处理人
   - 被派发人可以在小程序中查看并完成派给自己的报修

2. **接口更新**
   - 新增 `POST /repair/myReportedList`，查询我上报的报修
   - 新增 `POST /repair/myAssignedList`，查询派发给我的报修
   - 新增 `GET /repair/dispatchUsers/{companyId}`，查询当前单位可派发处理人
   - 新增 `POST /repair/dispatch`，管理员/项目负责人派发报修
   - `POST /repair/myList` 调整为“与我相关”的报修列表
   - `GET /repair/detail/{repairId}` 增加权限校验
   - `POST /repair/add` 会自动补齐报修人、单位、设备、系统等信息
   - `POST /repair/edit`、`POST /repair/delete/{repairId}` 改为报修人或管理员在待处理状态下可操作
   - `POST /repair/complete` 改为处理人或管理员可完成
   - `GET /repair/statistics` 新增 `reportedTotal`、`assignedTotal`

3. **数据结构调整**
   - 新增 `repairUserId`：处理人用户ID
   - 新增 `dispatchBy`：派发人
   - 新增 `dispatchTime`：派发时间
   - `repairStatus` 统一为：`0=待处理`、`1=处理中`、`2=已完成`

4. **小程序适配建议**
   - 报修列表建议拆分为“与我相关”“我上报的”“派给我的”3个页签
   - 提交报修时优先使用 `equipmentId`，由后端自动补齐设备名称、系统名称和位置
   - 管理员/项目负责人在详情页增加“派发”入口
   - 派发后，详情页展示派发人、派发时间、处理人
   - 完成报修页仅对处理人和管理员展示

### v1.4 (2024-01-20)
**新增功能：消防测试字段调整**

1. **字段使用层级调整**
   - 5个新增字段（设备位置、测试情况、测试时间、测试结果、现场照片）从第一级（检查项）调整到第二级（设备）
   - 第二级增加"维护"按钮，点击后填写这5个字段
   - 第一级（检查项）不再显示这5个字段

2. **接口更新**
   - 新增 `POST /task/updateMaintenance` 接口，用于更新第二级设备的维护信息
   - `POST /task/updateCheckDetail` 接口简化，移除了5个字段的参数
   - 第二级设备列表返回数据中包含这5个字段

3. **数据结构调整**
   - level=2（设备）：使用这5个字段
   - level=3（检查项）：不再使用这5个字段
   - 数据库字段保持不变，只是使用层级发生变化

**影响的接口：**
- `GET /task/system/{recordId}` - 返回的设备列表包含5个新字段
- `POST /task/updateMaintenance` - 新增接口，更新第二级设备维护信息
- `POST /task/updateCheckDetail` - 简化接口，移除5个字段参数

**小程序适配建议：**
- 在第二级设备列表页面增加"维护"按钮
- 点击"维护"按钮弹出表单，填写5个字段
- 第一级检查项详情页面移除这5个字段的输入
- 建议在设备卡片上显示是否已填写维护信息的标识

### v1.3 (2024-01-20)
**新增功能：常规维保与消防测试分离**

1. **任务类型区分**
   - 一个任务现在可以同时包含"常规维保"和"消防设施测试"两种类型
   - 通过 `recordType` 字段区分（0=常规维保，1=消防设施测试）
   - 小程序需要提供两个入口，分别进入不同类型的检查

2. **接口更新**
   - `GET /task/detail/{taskId}` 新增 `recordType` 查询参数
   - 支持按类型过滤系统列表
   - 返回数据中包含 `recordType` 字段

3. **页面结构调整**
   - 建议新增"任务操作选择页"，提供两个入口按钮
   - 常规维保和消防测试使用不同的主题色区分
   - 数据结构完全相同，都是三级分类

**新增字段：消防设施检查详情增强（已调整到第二级）**

为消防设施测试增加了5个新字段，用于更详细地记录设备测试信息。
**注意：这些字段现在在第二级（设备）使用，不再在第一级（检查项）使用。**

1. **deviceLocation** (设备位置)
   - 类型：String
   - 说明：记录设备的具体位置，如"1楼大厅"、"2楼东侧走廊"
   - 使用场景：便于快速定位设备
   - **使用层级：第二级（设备）**

2. **testSituation** (测试情况)
   - 类型：String (TEXT)
   - 说明：记录测试过程中的具体情况
   - 使用场景：详细记录测试过程
   - **使用层级：第二级（设备）**

3. **testTime** (测试时间)
   - 类型：DateTime
   - 格式：yyyy-MM-dd HH:mm:ss
   - 说明：记录具体的测试时间点
   - 使用场景：精确记录测试时间
   - **使用层级：第二级（设备）**

4. **testResult** (测试结果)
   - 类型：String (TEXT)
   - 说明：记录测试的详细结果
   - 使用场景：详细记录测试结果
   - **使用层级：第二级（设备）**

5. **sitePhotos** (现场照片)
   - 类型：String
   - 说明：存储现场照片的路径，多个照片用逗号分隔
   - 使用场景：提供现场的可视化证据
   - 注意：与 `faultImages`（故障图片）分开，`sitePhotos` 用于记录现场环境
   - **使用层级：第二级（设备）**

**影响的接口：**
- `GET /task/detail/{taskId}` - 新增 `recordType` 参数，返回数据包含 `recordType` 字段
- `GET /task/system/{recordId}` - 返回数据包含 `recordType` 字段，设备列表包含5个新字段
- `GET /task/equipment/{recordId}` - 返回数据中检查项不再包含5个新字段
- `POST /task/updateMaintenance` - 新增接口，更新第二级设备维护信息
- `POST /task/updateCheckDetail` - 简化接口，移除5个字段参数

**小程序适配建议：**
- 新增任务操作选择页，提供"常规维保"和"消防测试"两个入口
- 使用不同的主题色区分两种类型（蓝色/橙色）
- **在第二级设备列表页面增加"维护"按钮**
- **点击"维护"按钮弹出表单，填写5个字段（设备位置、测试情况、测试时间、测试结果、现场照片）**
- **第一级检查项详情页面移除这5个字段的输入**
- 现场照片建议使用图片上传组件，支持多图上传
- 测试时间建议使用日期时间选择器

## 三、检查操作接口

### 3.1 更新第二级设备维护信息（新增）

**接口地址:** `POST /task/updateMaintenance`

**请求方式:** POST (Content-Type: application/json)

**请求参数:**
```json
{
  "recordId": 201,
  "deviceLocation": "1楼配电室东侧",
  "testSituation": "设备运行正常，无异常声响",
  "testTime": "2024-01-20 14:30:00",
  "testResult": "压力测试通过，各项指标正常",
  "sitePhotos": "/profile/upload/2024/01/20/photo1.jpg,/profile/upload/2024/01/20/photo2.jpg"
}
```

**参数说明:**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| recordId | Long | 是 | 设备记录ID（第二级记录ID） |
| deviceLocation | String | 否 | 设备位置 |
| testSituation | String | 否 | 测试情况描述 |
| testTime | String | 否 | 测试时间（格式：yyyy-MM-dd HH:mm:ss） |
| testResult | String | 否 | 测试结果描述 |
| sitePhotos | String | 否 | 现场照片URL（多个用逗号分隔） |

**返回示例:**
```json
{
  "code": 0,
  "msg": "操作成功"
}
```

**说明:**
- 此接口用于更新第二级设备的维护信息
- 所有字段均为可选，只更新提交的字段
- 小程序在第二级设备列表页面增加"维护"按钮，点击后调用此接口

### 3.2 更新检查结果（快速操作）

**接口地址:** `POST /task/updateCheckResult`

**请求参数:**
```json
{
  "recordId": 301,
  "checkResult": "1"
}
```

**参数说明:**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| recordId | Long | 是 | 检查记录ID |
| checkResult | String | 是 | 检查结果（1=正常 2=故障 3=无此设备） |

**返回示例:**
```json
{
  "code": 0,
  "msg": "操作成功"
}
```

**说明:**
- 选择"正常"或"无此设备"时，会自动清空故障描述和故障图片
- 自动更新检查时间为当前时间
- 自动更新任务统计数据

### 6.3 注意事项

1. **图片上传流程**
   - 先调用 `/common/uploads` 上传图片
   - 获取返回的 `fileNames` 字段（相对路径）
   - 将 `fileNames` 作为 `faultImages` 或 `sitePhotos` 参数提交
   - 注意：`faultImages` 用于故障图片（第一级），`sitePhotos` 用于现场照片（第二级）

2. **现场照片与故障图片的区别（v1.4调整）**
   - `faultImages`: 故障图片，在第一级（检查项）使用，仅在设备状态为"故障"时使用，用于记录故障现象
   - `sitePhotos`: 现场照片，在第二级（设备）使用，通过"维护"按钮填写，用于记录现场环境和测试过程

3. **数据刷新**
   - 返回上一级时使用 `wx.redirectTo` 而不是 `wx.navigateBack`
   - 这样可以重新请求数据，刷新统计信息

4. **状态同步**
   - "是否完好"和"设备状态"需要保持同步
   - 选择"是"时，设备状态自动为"正常"
   - 选择"否"时，设备状态为"故障"或"无此设备"

5. **故障说明显示**
   - 只有选择"故障"状态时才显示故障说明输入框
   - 选择其他状态时自动隐藏并清空故障说明

6. **维护信息使用建议（v1.4调整）**
   - 维护信息在第二级（设备）填写，不在第一级（检查项）填写
   - 设备位置：建议提供常用位置的快速选择
   - 测试情况：可以提供模板文本供选择
   - 测试时间：默认为当前时间，可手动调整
   - 测试结果：可以提供常见结果的快速选择
   - 现场照片：建议限制单次最多上传5张，每张不超过2MB

7. **请求格式**
   - `/task/updateCheckDetail` 接口使用 JSON 格式（Content-Type: application/json）
   - `/task/updateMaintenance` 接口使用 JSON 格式（Content-Type: application/json）
   - 示例代码（第一级检查项）：
   ```javascript

     },
     success: (res) => {
       if (res.data.code === 0) {
   ```

   - 示例代码（第二级设备维护）：

### 8.2 JS逻辑示例

```javascript
Page({
  data: {
    recordId: 0,
    isGood: true,
    checkResult: '1',
    statusList: [
      { value: '1', label: '正常' },
      { value: '2', label: '故障' },
      { value: '3', label: '无此设备' }
    ],
    statusIndex: 0,
    faultDescription: '',
    deviceLocation: '',
    testSituation: '',
    testDate: '',
    testTime: '',
    testResult: '',
    sitePhotos: [],
    otherNotes: ''
  },

  onLoad(options) {
    this.setData({
      recordId: options.recordId
    });
    this.loadCheckDetail();
  },

  // 加载检查详情
  loadCheckDetail() {
    // 从上一页传递的数据或从服务器获取
    // ...
  },

  // 是否完好改变
  onGoodChange(e) {
    const isGood = e.detail.value === 'yes';
    this.setData({
      isGood: isGood,
      checkResult: isGood ? '1' : '2',
      statusIndex: isGood ? 0 : 1
    });
  },

  // 设备状态改变
  onStatusChange(e) {
    const index = e.detail.value;
    const status = this.data.statusList[index];
    this.setData({
      statusIndex: index,
      checkResult: status.value,
      isGood: status.value === '1'
    });
  },

  // 选择照片
  choosePhoto() {
    const that = this;
    wx.chooseImage({
      count: 5 - this.data.sitePhotos.length,
      sizeType: ['compressed'],
      sourceType: ['album', 'camera'],
      success(res) {
        that.uploadPhotos(res.tempFilePaths);
      }
    });
  },

  // 上传照片
  uploadPhotos(tempFilePaths) {
    const that = this;
    wx.showLoading({ title: '上传中...' });
    
    const uploadPromises = tempFilePaths.map(filePath => {
      return new Promise((resolve, reject) => {
        wx.uploadFile({
          url: 'http://your-domain.com/common/upload',
          filePath: filePath,
          name: 'file',
          success(res) {
            const data = JSON.parse(res.data);
            if (data.code === 0) {
              resolve(data.fileName);
            } else {
              reject(data.msg);
            }
          },
          fail: reject
        });
      });
    });

    Promise.all(uploadPromises)
      .then(fileNames => {
        const sitePhotos = [...that.data.sitePhotos, ...fileNames];
        that.setData({ sitePhotos });
        wx.hideLoading();
        wx.showToast({ title: '上传成功' });
      })
      .catch(err => {
        wx.hideLoading();
        wx.showToast({ title: '上传失败', icon: 'none' });
      });
  },

  // 删除照片
  deletePhoto(e) {
    const index = e.currentTarget.dataset.index;
    const sitePhotos = this.data.sitePhotos;
    sitePhotos.splice(index, 1);
    this.setData({ sitePhotos });
  },

  // 预览照片
  previewPhoto(e) {
    const url = e.currentTarget.dataset.url;
    const urls = this.data.sitePhotos.map(photo => 
      'http://your-domain.com' + photo
    );
    wx.previewImage({
      current: 'http://your-domain.com' + url,
      urls: urls
    });
  },

  // 提交表单
  onSubmit() {
    const data = {
      recordId: this.data.recordId,
      checkResult: this.data.checkResult,
      faultDescription: this.data.faultDescription,
      deviceLocation: this.data.deviceLocation,
      testSituation: this.data.testSituation,
      testTime: this.data.testDate && this.data.testTime ? 
        `${this.data.testDate} ${this.data.testTime}:00` : '',
      testResult: this.data.testResult,
      sitePhotos: this.data.sitePhotos.join(','),
      otherNotes: this.data.otherNotes
    };

    wx.request({
      url: 'http://your-domain.com/api/fire/task/updateCheckDetail',
      method: 'POST',
      header: {
        'content-type': 'application/json'
      },
      data: data,
      success: (res) => {
        if (res.data.code === 0) {
          wx.showToast({ title: '保存成功' });
          setTimeout(() => {
            wx.navigateBack();
          }, 1500);
        } else {
          wx.showToast({ 
            title: res.data.msg || '保存失败', 
            icon: 'none' 
          });
        }
      },
      fail: () => {
        wx.showToast({ title: '网络错误', icon: 'none' });
      }
    });
  },

  onCancel() {
    wx.navigateBack();
  }
});
```

2. 自动填充设备信息
3. 填写故障描述和上传图片

**场景3：设备档案查询**
1. 扫描二维码查看设备完整档案
2. 包括：基本信息、维保记录、故障历史

### 12.1 页面结构建议

```
pages/
```

### 12.2 开发流程

**1. 设备列表页**
- 调用 `POST /equipment/list` 获取设备列表
- 支持按公司、建筑、设备类型筛选
- 支持设备名称和编码搜索
- 显示设备状态标识（正常/故障/停用）
- 点击设备进入详情页

**2. 设备详情页**
- 调用 `GET /equipment/detail/{equipmentId}` 获取详情
- 显示设备完整信息
- 提供编辑、报修、巡检入口
- 显示维保历史记录

**3. 扫码功能**
- 使用 `wx.scanCode()` 扫描二维码
- 解析二维码获取设备编码
- 调用 `GET /equipment/scan/{equipmentCode}` 获取设备信息
- 跳转到设备详情页或操作页面

**4. 新增/编辑设备**
- 选择公司和建筑
- 填写设备基本信息
- 调用 `POST /equipment/add` 或 `POST /equipment/edit`
- 保存成功后返回列表页

### 12.3 扫码示例代码

```javascript
// 扫描设备二维码
scanEquipment() {
  wx.scanCode({
    success: (res) => {
      // 解析二维码内容，提取设备编码
      const url = res.result;
      const equipmentCode = url.substring(url.lastIndexOf('/') + 1);
      
      // 调用接口获取设备信息
      wx.request({
        url: 'http://your-domain.com/api/fire/equipment/scan/' + equipmentCode,
        method: 'GET',
        success: (res) => {
          if (res.data.code === 0) {
            // 跳转到设备详情页
            wx.navigateTo({
              url: '/pages/equipment/detail?equipmentId=' + res.data.data.equipmentId
            });
          } else {
            wx.showToast({
              title: '设备不存在',
              icon: 'none'
            });
          }
        }
      });
    }
  });
}
```

### 12.4 注意事项

1. **权限控制**
   - 普通用户只能查看设备信息
   - 维保人员可以编辑设备状态
   - 管理员可以新增/删除设备

2. **数据缓存**
   - 设备列表可缓存5分钟
   - 设备详情实时获取
   - 扫码结果不缓存

3. **离线支持**
   - 扫码后缓存设备信息
   - 离线时可查看已缓存的设备
   - 网络恢复后同步数据

4. **图片处理**
   - 设备图片建议压缩后上传
   - 单张图片不超过2MB
   - 支持多张图片上传

## 十六、故障报修管理接口（v1.5）

### 16.1 功能说明

故障报修模块支持两种业务场景：

1. **小程序主动上报**
   - 现场人员在小程序中主动提交故障报修
   - 报修人可以持续查看自己提交的报修
   - 管理员/项目负责人可以查看当前单位下全部报修

2. **管理员派发处理**
   - 管理员/项目负责人选择当前单位的处理人进行派发
   - 被派发人可以在小程序中查看“派给我的”报修并完成处理
   - 对于未完成的报修，管理员可以再次派发给其他处理人

### 16.2 权限与可见范围

| 角色 | 可查看范围 | 可操作范围 |
|------|------------|------------|

| 项目负责人（`roleType=1`） | 当前单位全部报修 | 编辑/删除当前单位待处理报修、派发、重新派发、完成处理中报修 |
| 管理员（`roleType=2`） | 当前单位全部报修 | 同项目负责人 |
| 系统管理员 | 全部 | 全部 |

**补充说明：**

```json
  "faultDescription": "阀门漏水，地面有积水",
  "faultImages": "/profile/upload/2026/03/16/a.jpg,/profile/upload/2026/03/16/b.jpg",
  "customerAddress": "1层东侧走廊",
  "remark": "现场已经做临时围挡"
}
```

**参数说明：**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| companyId | Long | 是 | 单位ID |
| systemTypeId | Long | 否 | 系统类型ID，传 `equipmentId` 时可不传 |
| equipmentId | Long | 否 | 设备ID，建议传 |
| urgencyLevel | String | 否 | 紧急程度，默认一般 |
| isReported | String | 否 | 是否已向客户/上级上报，默认 `1` |
| foundTime | DateTime | 否 | 发现时间，不传则取当前时间 |
| faultDescription | String | 是 | 故障描述 |
| faultImages | String | 否 | 故障图片 |
| customerAddress | String | 否 | 客户地址/故障位置 |
| remark | String | 否 | 备注 |

**后端自动处理：**
- 自动补齐 `reporterId`、`reporterName`、`reporterPhone`
- 自动生成 `repairNo`
- 自动补齐 `companyName`
- 若传了 `equipmentId`，会自动补齐 `equipmentName`、`systemTypeId`、`systemTypeName`、`customerAddress`
- 默认状态为 `repairStatus=0`

**返回示例：**
```json
{
  "code": 0,
  "msg": "报修提交成功",
  "data": {
    "repairId": 12,
    "repairNo": "BX202603160001"
  }
}
```

**说明：**

- 报修人本人、项目负责人、管理员可删除

#### 16.6.5 派发报修

**接口地址：** `POST /repair/dispatch`

**说明：**
- 仅管理员/项目负责人可调用
- 可对待处理、处理中报修进行派发或改派
- 已完成报修不可派发

**请求参数：**
```json
{
  "repairId": 12,
  "repairUserId": 305
}
```

**派发后的结果：**
- `repairStatus` 更新为 `1`
- 自动写入 `repairUserId`
- 自动写入 `repairPerson`、`repairPhone`
- 自动写入 `dispatchBy`、`dispatchTime`、`acceptTime`

#### 16.6.6 兼容旧受理接口

**接口地址：** `POST /repair/accept`

**说明：**
- 推荐新项目优先使用 `POST /repair/dispatch`
- 如果请求体中包含 `repairUserId`，则按“派发”处理
- 如果只传 `repairPerson`、`repairPhone`，则按旧逻辑手工受理

**请求示例1：按新逻辑派发**
```json
{
  "repairId": 12,
  "repairUserId": 305
}
```

**请求示例2：按旧逻辑手工受理**
```json
{
  "repairId": 12,
  "repairPerson": "李四",
  "repairPhone": "13900139000"
}
```

**说明：**

### 16.8 推荐接入流程

#### 16.8.1 普通用户上报流程

1. 调用 `/company/current` 或前端保存当前单位
2. 调用 `/systemType/byCompany/{companyId}` 获取系统类型
3. 调用 `/equipment/byCompanyAndSystem` 获取设备
4. 调用 `/repair/add` 提交报修
5. 调用 `/repair/myReportedList` 或 `/repair/myList` 查看进度
6. 点击报修记录时调用 `/repair/detail/{repairId}`

#### 16.8.2 管理员派发流程

1. 调用 `/repair/list` 查看当前单位报修
2. 调用 `/repair/dispatchUsers/{companyId}` 获取处理人
3. 调用 `/repair/dispatch` 完成派发
4. 处理人进入 `/repair/myAssignedList` 查看待处理报修
5. 处理完成后调用 `/repair/complete`

#### 16.8.3 小程序页面建议

- 报修首页：统计卡片调用 `/repair/statistics`
- 列表页：使用 3 个页签
  - 与我相关：`/repair/myList`
  - 我上报的：`/repair/myReportedList`
  - 派给我的：`/repair/myAssignedList`
- 详情页：展示报修信息、派发信息、处理信息、时间线
- 管理员详情页：增加“派发/改派”按钮
- 完成页：仅在 `repairStatus=1` 且当前用户是处理人/管理员时显示

### 16.9 与维保简报的关系

- 已完成的报修记录仍可通过 `GET /repair/completed/{companyId}` 查询
- 该接口主要用于维保简报和报告引用
- 如果小程序仅做报修业务，优先使用本章中的 `/repair/list`、`/repair/detail`、`/repair/statistics`
