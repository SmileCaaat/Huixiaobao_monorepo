# 核心业务流程

## 1. 登录和会话

```text
pages/login/index
→ pages/login/password
→ POST /api/login {username,password}
→ Shiro Subject.login
→ 返回 Session ID 作为 token
→ 本地 storage 保存 token
→ reLaunch pages/index/index
→ 后续请求携带 Authorization: Bearer <token>
```

后端 Session 默认空闲 30 分钟。无 refresh token。退出时调用 `/api/logout`，随后删除本地 Token。

注意：当前真实 HTTP 401 不会被前端正确处理；原生重构前应先修复统一请求层。

## 2. 公司上下文

公司是多数现场业务的上层上下文：

```text
GET /api/fire/company/myList
→ 用户选择 companyId
→ POST /api/fire/company/switch
→ 后端保存 currentCompanyId 到 Shiro Session
→ GET /api/fire/company/current
→ 各页面携带或读取 companyId
```

后端部分接口还会从 Session 中读取 `currentCompanyId`，但建筑、设备、巡检等接口也接受客户端 `companyId`。应在后端统一验证当前用户是否能访问该公司。

## 3. 维保任务

### 页面链路

```text
首页
→ task/index：我的任务
→ task/action：常规维保 / 消防设施测试
→ task/detail：一级系统列表
→ task/system：二级设备列表
→ task/device：三级检查项
→ 更新检查结果、故障描述、图片和其他说明
```

### API 链路

```text
GET  /api/fire/company/current
POST /api/fire/task/myList
GET  /api/fire/task/detail/{taskId}
GET  /api/fire/task/system/{recordId}
GET  /api/fire/task/equipment/{recordId}
POST /api/fire/task/updateCheckResult
POST /api/fire/task/updateFaultDesc
POST /api/fire/task/updateCheckDetail
```

关键标识：

- `companyId`：客户/项目范围。
- `taskId`：维保任务。
- `recordId`：系统、设备或检查项记录。
- `recordType`：`0` 常规维保，`1` 消防设施测试。

### 完成规则

小程序当前没有整单“提交完成”动作。后端在任务列表查询中统计所有 `level=3` 的检查项：只要全部 `check_result != 0`，列表状态即表现为完成。

### 当前断点

二级设备的维护/测试弹窗提交到 `/api/fire/task/updateMaintenance`，后端没有该 REST 路径。管理端同名接口不能直接替代。

## 4. 设备管理

```text
GET 当前公司
→ POST /api/fire/equipment/list
→ 前端按 systemName 分组
→ 类型列表
→ GET /api/fire/equipment/detail/{equipmentId}
→ 新增或编辑
→ POST /api/common/upload
→ POST /api/fire/equipment/add|edit
→ 刷新设备列表
```

核心字段：`equipmentId`、`equipmentCode`、`companyId`、`buildingId`、`systemTypeId`、`systemName`、`equipmentName`、`location`、`expireDate`、`image`。

设备二维码扫描后把扫描值作为 `equipmentCode` 查询。如果二维码内容不是纯设备编码，原生重构时需要明确二维码格式协议。

## 5. 巡检登记

```text
GET 当前公司
→ POST /api/fire/inspection/myList
→ 新建巡检
→ 选择建筑、系统并录入设备状态
→ 上传照片
→ POST /api/fire/inspection/add
→ 后端写入当前 inspectorId/inspectorName
→ 返回 inspectionId
```

主要字段：`companyId`、`buildingId`、`inspectionType`、`systemType`、`equipmentName`、`equipmentCount`、`inspectionTime`、`equipmentStatus`、`imageUrls`。

## 6. 维保签到

```text
获取当前公司或由 taskId 反查任务
→ wx.getLocation 获取 GCJ-02 坐标
→ 获取公司任务列表
→ 选择签到/签退
→ 上传至少一张图片
→ POST /api/fire/checkIn/add
→ 后端计算与公司签到点距离并保存
```

关键字段：`companyId`、`taskId`、`checkInType`、`latitude`、`longitude`、`address`、`images`。

后端即使超出签到范围也会保存签到，并在响应中返回 `isInRange` 和 `distance`。是否允许范围外签到属于业务规则，需要产品或运维确认。

## 7. 故障报修

```text
选择当前公司、系统和可选设备
→ 上传故障图片
→ POST /api/fire/repair/add
→ 我上报的 / 派发给我的
→ 管理员或项目负责人 dispatch（必选处理人 repairUserId）
→ 未开工可 recall 撤回派发
→ 处理人 complete
```

状态：

- `0`：待处理（未派发，或撤回后恢复）。
- `1`：处理中（已派发）。
- `2`：已完成。

关键字段：`repairUserId` / `repairPerson` / `dispatchBy` / `dispatchTime` / `startTime`。

管理端撤回：`POST /fire/repair/recall`（参数 `repairId`）。  
仅已派发、有处理人、尚未开始处理（无 `startTime` / 维修说明 / 图片）、未完成时可撤回；成功后清空处理人与派发信息，状态回 `0`。权限复用 `fire:repair:accept`。

小程序 `myAssignedList` 按 `repair_user_id` 过滤，撤回后原处理人不再可见，完成接口也会拒绝已撤回工单。

## 8. 报告

```text
管理端：选择客户 → 选择该客户维保任务 → POST /fire/report/generate
→ 生成 PDF 至 uploadPath/report/
→ 列表预览 GET /fire/report/view/{reportId}
→ 或下载 GET /fire/report/download/{reportId}

小程序：
POST /api/fire/report/list
→ GET /api/fire/report/detail/{reportId}
→ GET /api/fire/report/preview/{reportId}
或 GET /api/fire/report/download/{reportId}
```

- 新报告默认输出 PDF；历史 DOCX 可下载，不可伪造成 PDF 在线预览，需重新生成。
- 预览/下载使用相对路径与登录会话，不依赖写死的 `127.0.0.1` 或固定端口。
- 生产域名需配置为微信下载合法域名（小程序端）。
