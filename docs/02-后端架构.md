# 后端架构

## 1. 启动和配置

启动类位于 `ruoyi-admin`：

```text
backend/ruoyi-admin/src/main/java/com/ruoyi/RuoYiApplication.java
```

主要配置：

| 文件 | 作用 |
|---|---|
| `application.yml` | 端口、上传、MyBatis、PageHelper、Shiro、XSS/CSRF、Swagger |
| `application-druid.yml` | MySQL、Druid 连接池和监控控制台 |
| `logback.xml` | 日志配置 |
| `mybatis/mybatis-config.xml` | MyBatis 全局配置 |
| `ehcache/ehcache-shiro.xml` | Shiro 缓存 |

关键运行参数：

- `server.port=83`
- `ruoyi.profile=./uploadPath`
- `spring.profiles.active=druid`
- `DB_URL`：数据库 URL；代码中存在外部地址默认值，本地和生产都建议显式覆盖。
- `DB_USER`：数据库用户，默认 `root`。
- `DB_PASS`：无默认值，启动必须提供。
- `DRUID_USER`、`DRUID_PASS`：Druid 控制台账号。

父 POM 写 Java 8，但 `ruoyi-system/pom.xml` 强制 Java 21 preview，Docker 也使用 Temurin 21。因此开发、构建和运行统一采用 JDK 21。

## 2. 分层结构

消防业务调用链：

```text
Controller
  ruoyi-admin/com.ruoyi.web.controller.fire       管理端 HTML/表单
  ruoyi-admin/com.ruoyi.web.controller.api        小程序 JSON API
        ↓
Service interface
  ruoyi-system/com.ruoyi.fire.service
        ↓
Service implementation
  ruoyi-system/com.ruoyi.fire.service.impl
        ↓
Mapper interface
  ruoyi-system/com.ruoyi.fire.mapper
        ↓
Mapper XML
  ruoyi-system/src/main/resources/mapper/fire
        ↓
MySQL fire_* tables
```

管理端通常按业务拆 Controller；小程序接口主要集中在 `FireMiniAppController`。后者超过一千行，后续后端维护可按 company/building/equipment/task/inspection/repair/report 拆成多个 REST Controller，但拆分时应保持 URL 兼容。

## 3. 消防业务包

### 核心实体

| 领域 | 实体 |
|---|---|
| 客户 | `FireCompany`、`FireUserCompany`、`FireContract` |
| 建筑与设备 | `FireBuilding`、`FireEquipment`、`FireSystemType` |
| 维保任务 | `FireMaintenanceTask`、`FireMaintenanceTemplate`、`FireMaintenanceRecord` |
| 任务层级辅助 | `FireMaintenanceSystem`、`FireMaintenanceDevice`、`FireMaintenanceItem` |
| 签到 | `FireCheckIn`、`FireCheckInImage` |
| 巡检 | `FireInspection`、`FireInspectionSession` |
| 报修 | `FireFaultRepair` |
| 报告 | `FireReportRecord`，当前位于 `com.ruoyi.system.domain` 而非 `com.ruoyi.fire.domain` |

### 任务数据模型

任务使用三级记录结构：

```text
FireMaintenanceTask
└─ FireMaintenanceRecord level=1  系统
   └─ FireMaintenanceRecord level=2  设备
      └─ FireMaintenanceRecord level=3  检查项
```

`recordType` 区分：

- `0`：常规维保。
- `1`：消防设施测试。

`checkResult` 区分：

- `0`：待检查。
- `1`：正常。
- `2`：故障。
- `3`：无此设备。

任务列表状态并不完全依赖 `task_status` 字段。Mapper 会统计三级检查项，当所有三级项均已处理时动态返回完成状态。

## 4. 认证与权限

### 管理端

- Shiro Realm 完成账号密码认证。
- 角色和菜单权限来自 RuoYi 系统表。
- 管理端使用 Cookie/Session、Shiro 权限注解和菜单权限。

### 小程序

小程序 Token 实际是 Shiro Session ID，不是 JWT：

1. `POST /api/login` 创建 `UsernamePasswordToken`。
2. 登录成功后返回 `subject.getSession().getId()`。
3. 小程序发送 `Authorization: Bearer <sessionId>`。
4. `OnlineWebSessionManager.getSessionId()` 从 Header 还原 Session。
5. `/api/**` 由 `ApiTokenFilter` 校验是否已认证。

匿名接口：

- `/api/login`
- `/api/register`
- `/api/fire/dept/list`

Session 默认空闲超时 30 分钟。没有 refresh token 或专门的续期接口。

### 数据权限现状

任务和报修的部分接口已经做了“当前用户是否为任务负责人/执行人/操作员”和公司权限判断。公司详情、建筑、设备、巡检等若干小程序接口仍缺少一致的资源级权限校验，重构前应形成统一的 company scope 校验方法。

| 文件 | 内容 |

```text

├─ 001_ruoyi_schema.sql
├─ 002_fire_schema.sql
├─ 003_quartz_schema.sql
└─ 004_seed_required_dicts.sql
```

## 6. 文件、报告和基础设施

### 上传

- 小程序接口：`POST /api/common/upload`
- 管理端公共接口也存在上传能力。
- 文件保存到 `./uploadPath/upload/...`。
- 对外访问前缀为 `/profile/**`。
- 单文件限制 10 MB，总请求 20 MB。
- Docker 部署需持久化 `/app/uploadPath`。

### 报告

- 报告元数据来自 `fire_report_record`（`reportName` / `filePath` / `fileSize`）。
- **最终产物为 PDF**（模板仍为 DOCX：`poi-tl` 填充 → LibreOffice 转 PDF）。
- 文件统一落在 `ruoyi.profile/report/`，即 `./uploadPath/report/`；库中只存文件名，禁止路径穿越。
- 管理端：
  - 手动生成：`POST /fire/report/generate`（必填 `companyId` + `taskId`，后端校验任务归属客户）
  - 客户/任务联动：`GET /fire/report/companies`、`GET /fire/report/tasks?companyId=`
  - 预览页：`GET /fire/report/view/{reportId}`（相对路由，不写死域名端口）
  - 流式预览/下载：`GET /fire/report/preview/{reportId}`、`GET /fire/report/download/{reportId}`（`application/pdf`）
- 小程序同样走 `/api/fire/report/preview|download/{reportId}`，按记录解析 PDF；历史 DOCX 不可当作 PDF 预览。
- 部署需安装 **LibreOffice**（或配置 `ruoyi.report.libreOfficePath`），并持久化 `uploadPath`。

### 定时任务

- 管理入口：`/monitor/job`、`/monitor/jobLog`。
- 调度配置类当前整体注释；单机模式主要依赖 Spring Boot Quartz 默认配置。
- `quartz.sql` 可用于数据库 JobStore/集群，但不能仅因脚本存在就认定生产已经启用数据库集群调度。

### 代码生成器

- 路径：`/tool/gen`。
- 基于 MySQL 元数据和 Velocity。
- 默认包名仍是 `com.ruoyi.system`，为消防模块生成代码前需检查 `generator.yml`。
