# 开发与部署

## 1. 后端本地开发

> 更省事的一键联调见 [`09-local_sever.md`](09-local_sever.md)（`local_sever/start.bat`）。

### 依赖

- JDK 21
- Maven 3.x
- MySQL
- 可写的 `backend/uploadPath`

未发现 Redis、消息队列等必需中间件。

### 环境变量

PowerShell 示例：

```powershell
cd C:\Users\JamLew\Desktop\Huixiaobao\backend

$env:DB_URL = "jdbc:mysql://127.0.0.1:3306/dev_manager?useUnicode=true&characterEncoding=utf8&serverTimezone=GMT%2B8"
$env:DB_USER = "root"
$env:DB_PASS = "replace-me"
$env:DRUID_USER = "admin"
$env:DRUID_PASS = "replace-me"
```

不要把真实密码提交到仓库。

### 构建和启动

```powershell
mvn clean package -DskipTests
java -jar ruoyi-admin\target\ruoyi-admin.jar
```

默认地址：

```text
http://localhost:83/
```

也可以在 IDE 中直接运行 `com.ruoyi.RuoYiApplication`。

### 数据库初始化限制

`ry_20250416.sql` 只提供 RuoYi 系统基线；仓库缺少完整的消防业务建表脚本。因此全新环境启动前必须从可信环境取得 `fire_*` 表结构。不要把现有增量脚本当作完整初始化脚本直接按文件名全部执行。

## 2. 小程序本地开发

### 打开项目

1. 打开微信开发者工具。
2. 选择“导入项目”。
3. 项目目录选择仓库下的 `miniprogram/`。
4. `project.config.json` 已包含 AppID 和基础库版本。
5. 当前 `urlCheck=true`，后端域名必须在微信公众平台配置为合法 request/upload/download 域名。

### 当前限制

- 该目录是编译产物，无需 npm 构建。
- 缺少原始 uni-app 工程，不能安全地通过 HBuilderX/CLI 重编译。
- 直接修改 `.js/.wxml/.wxss/.json` 只能作为临时维护方式。

### 本地后端联调

当前 Base URL 硬编码为线上 HTTPS 域名。建议先把它改成环境配置，而不是每次手改：

```javascript
const ENV = "dev"
const BASE_URLS = {
  dev: "http://127.0.0.1:83",
  prod: "https://huixiaobao-admin.site"
}
```

手机真机不能通过 `127.0.0.1` 访问电脑，需要局域网地址、代理或测试域名。

## 3. 生产手工更新

根目录 PDF 记录的现有流程如下：

1. 本地使用 JDK 21 和 Maven。
2. Maven 先执行 `clean`，再执行 `package`。
3. 产物位于 `backend/ruoyi-admin/target/ruoyi-admin.jar`。
4. 把 JAR 上传到服务器 `/opt/app`。
5. 在 `/opt/app` 执行：

```sh
sh restart-ruoyi.sh
```

6. 查看启动日志：

```sh
tail -f /opt/app/nohup.out
```

原始来源：[`汇消宝管理后台更新部署文档.pdf`](../汇消宝管理后台更新部署文档.pdf)。

`restart-ruoyi.sh` 不在当前仓库中，只存在于生产服务器。发布前必须先读取该脚本，确认它使用的 JAR 名称、Java 路径、环境变量、端口和停止策略。

## 4. 建议补全的手工发布流程

现有 PDF 没有备份、校验和回滚步骤。建议标准化为：

```sh
cd /opt/app

# 1. 确认当前进程和磁盘
ps -ef | grep ruoyi-admin.jar
df -h

# 2. 备份当前包
cp ruoyi-admin.jar backup/ruoyi-admin-YYYYMMDD-HHMMSS.jar

# 3. 上传到临时文件名并校验
# ruoyi-admin.jar.new
sha256sum ruoyi-admin.jar.new

# 4. 原子替换
mv ruoyi-admin.jar.new ruoyi-admin.jar

# 5. 重启
sh restart-ruoyi.sh

# 6. 观察日志和健康检查
tail -n 200 -f /opt/app/nohup.out
curl -I http://127.0.0.1:<port>/login
```

真正执行前应由运维确认脚本和服务端约定；上面是建议模板，不代表生产当前已经按此配置。

### 回滚建议

```sh
cp backup/ruoyi-admin-<timestamp>.jar ruoyi-admin.jar
sh restart-ruoyi.sh
tail -n 200 -f /opt/app/nohup.out
```

涉及数据库迁移时，必须单独准备可回滚脚本或确认迁移向后兼容；不能只回滚 JAR。

## 5. Docker 路径

仓库提供两阶段 Dockerfile：

```powershell
docker build -t huixiaobao-backend .\backend

docker run --rm `
  -p 83:80 `
  -e DB_URL="jdbc:mysql://host:3306/dev_manager?..." `
  -e DB_USER="root" `
  -e DB_PASS="replace-me" `
  -v C:\data\huixiaobao-upload:/app/uploadPath `
  huixiaobao-backend
```

当前没有 Docker Compose，也没有证据表明生产使用 Docker；不能把 Dockerfile 与 PDF 中的 `/opt/app` 手工 JAR 部署混为一谈。

## 6. 发布前检查

- 确认目标分支和 commit。
- 确认 JDK 21/Maven 构建通过。
- 检查是否包含数据库脚本以及执行顺序。
- 备份旧 JAR。
- 备份或确认 `uploadPath` 持久化。
- 确认 `DB_URL/DB_USER/DB_PASS` 来源。
- 确认 Nginx/HTTPS 和小程序合法域名。
- 重启后检查登录、公司切换、任务列表、设备列表、上传和报告下载。
- 观察 `nohup.out` 和应用日志是否有数据库、端口、权限或文件路径异常。
