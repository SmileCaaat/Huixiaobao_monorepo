# 小程序架构与现状

## 1. 最重要的结论

`miniprogram/` 是 uni-app 编译到微信小程序后的产物，不是可持续维护的源工程。对它继续直接修改可以临时修线上问题，但任何来自原始 uni-app 工程的重新编译都可能覆盖这些修改。

证据：

- `app.js` 使用 `createSSRApp`、`common_vendor`。
- 所有 31 个页面 JS 均有 `wx.createPage` 和 uni-app 编译包装。
- 22 个页面直接依赖 `node-modules/@dcloudio/uni-ui`。
- `common/vendor.js` 约 278 KB。
- JS 中保留 `.vue` 源行号和 `sourceMappingURL`，但仓库没有 sourcemap。
- 没有 `.vue`、`pages.json`、`manifest.json`、`package.json`。
- `改动记录.md` 明确称其为编译产物。

因此整体原生重构是合理选择。核心原则是保留后端业务和 URL 契约，替换小程序表现层、状态管理和请求基础设施。

## 2. 入口和路由

`app.json` 注册 31 个页面，首屏为 `pages/login/index`。没有原生 `tabBar` 配置，项目使用自定义 `components/TabBar`，并通过 `redirectTo` 在首页、消息、扫码、个人中心之间切换。

主要路由：

```text
login/index
├─ login/password
└─ login/register

index/index
├─ checkin/index → checkin/history
├─ task/index → task/action → task/detail → task/system → task/device
├─ customer/index
├─ equipment/index → equipment/list → equipment/detail → equipment/edit
│                                   └→ equipment/form
├─ report/index
├─ building/index → building/detail / building/form
├─ repair/form
└─ repair/index → repair/detail
```

`inspection` 页面仍存在，但首页入口目前被注释。

## 3. 目录职责

| 目录 | 职责 |
|---|---|
| `api/` | 封装全部后端路径，对页面暴露业务函数 |
| `utils/request.js` | Base URL、Token、GET/POST、统一 loading/error、上传 |
| `common/vendor.js` | Vue/uni-app 运行时和 `uni.*` 到微信 API 的适配 |
| `components/CompanyDrawer` | 当前用户公司列表、搜索和切换公司 |
| `components/TabBar` | 自定义底部导航 |
| `node-modules/@dcloudio/uni-ui` | 编译后 uni-ui 组件 |
| `pages/` | 页面逻辑、视图和页面配置 |

## 4. 请求与 Token

当前请求链：

```text
页面
→ api/index.js
→ utils/request.js
→ https://huixiaobao-admin.site
→ Authorization: Bearer <token>
```

Token：

- key：`token`
- 存储：微信同步 storage
- 来源：`POST /api/login` 返回的 Shiro Session ID
- 登出：调用 `/api/logout` 后移除本地 Token

### 已知问题

1. `BASE_URL` 硬编码，缺少 dev/test/prod 环境配置。
2. 后端认证失败返回 HTTP 401；前端只在 HTTP 200 内判断 body `code=401`，所以真实过期不会触发清 Token 和重登。
3. 普通请求封装与多个页面重复实现上传，返回字段归一化不一致。
4. 部分页面自行 `showLoading/hideLoading`，请求层也会显示 loading，容易重复。
5. 下载/预览报告自行拼接 URL 和 Header，没有统一下载模块。

## 5. 页面模块现状

| 模块 | 当前作用 | 主要问题 |
|---|---|---|
| `login` | 密码登录、注册 | 手机验证码页无真实发送接口且引用未导入变量；协议页面未注册 |
| `index` | 公司、应用入口、统计 | 首页统计调用管理端 HTML 路由，失败后展示硬编码演示数据 |
| `task` | 任务三级执行 | 分页/筛选契约问题；`updateMaintenance` 后端 API 缺失 |
| `equipment` | 分组、列表、详情、表单 | 字段别名多；表单重度依赖 uni-forms/uni-popup |
| `inspection` | 列表、登记、详情 | 首页入口注释；复杂表单重度依赖 uni-ui |
| `checkin` | 定位、签到/签退、图片 | 重复上传实现；位置校验接口未在提交前单独调用 |
| `repair` | 报修、派发、完成 | 整体接口相对清晰，适合作为原生复杂业务参考模块 |
| `building` | 建筑 CRUD | 表单和上传仍为编译代码 |
| `report` | 报告列表和下载 | 依赖后端文件目录和下载合法域名 |
| `message` | 消息 | 基本为空，暂不应优先迁移 |

## 6. 字段漂移

设备页面为了兼容历史响应，同时处理：

- `equipmentType` / `systemName` / `systemType`
- `expireDate` / `expiryDate` / `warrantyEndDate`
- `model` / `specification` / `specifications`
- `floorNo` / `floor`
- `image` / `imageUrls`

原生重构时不应继续扩大别名集合。建议以 Java Domain JSON 字段为准，在 API 适配层做一次性归一化，并为每个接口记录请求/响应样例。

## 7. 原生重构目标结构

建议的新结构：

```text
miniprogram/
├─ app.js
├─ app.json
├─ app.wxss
├─ config/
│  ├─ env.js
│  └─ constants.js
├─ services/
│  ├─ request.js
│  ├─ auth.js
│  ├─ company.js
│  ├─ task.js
│  ├─ equipment.js
│  └─ ...
├─ models/
│  └─ normalizers.js
├─ components/
├─ pages/
└─ utils/
```

重构完成标准：

- 不再引用 `common/vendor.js` 和 `node-modules/@dcloudio`。
- 页面使用原生 `Page`/`Component` 和 `wx.*`。
- 请求、上传、下载和 401 仅有一个实现。
- Base URL 可按环境切换。
- 页面不再使用硬编码演示业务数据掩盖接口错误。
- 核心流程具备最少的接口冒烟测试或可重复人工验收脚本。
